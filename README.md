## 4. Git, GitHubに関する調査

### 4-1. Gitを始めとしたバージョン管理システムを利用することによるメリットはなにか
バージョン管理システム(VCS)を利用することで得られる代表的なメリットとして次の3点が挙げられる。

1. **変更の記録と追跡ができる**  
   共同作業では、チームメンバーによってコードやドキュメントに変更が加わる。このようなときにVCSでは、ファイルの変更履歴を詳細に記録し、誰が、いつ、どのような変更を行ったかを追跡できる。

2. **リスクの軽減が可能**  
   新しい機能の追加や既存のコードの変更によってコードが正しく動作しなくなることがある。VCSでは、もし新しい変更が問題を引き起こした場合でも、簡単に以前の安定した状態に戻すことができる。これにより、開発者は安心して開発を進めることができる。

3. **効率的な共同開発が可能**  
   VCSは、異なる開発者が同じファイルに同時に作業する際の衝突を防ぎ、変更内容を効率的に統合することができる。これにより、スムーズな共同開発が可能になる。

### 4-2. Gitにおけるキーワード「clone」「commit」「push」「branch」「fetch」「pull」「merge」「rebase」についてそれぞれ何をするものか

#### clone
既存のリポジトリをローカル環境に複製するもの[1]。

#### commit
Gitのリポジトリに作業内容（ファイルの変更など）をセーブするもの。コミットを利用することで、自分の作業履歴を記録し、以前の状態を保持（戻す）することが可能になる[2]。

#### push
ローカルリポジトリの変更をリモートリポジトリへ反映するもの。pushを実行すると、リモートリポジトリに自分の変更履歴がアップロードされ、他メンバーと変更履歴を共有できるようになる[3]ｌ

#### branch
プロジェクト内で作業の流れを分岐するもの。これにより、メインのコードベースに影響を与えずに新機能の開発やバグ修正を行うことが可能となる[4]。

#### fetch
リモートリポジトリの最新の変更を取得し、ローカルのリモート追跡ブランチに反映するもの。ただし、現在の作業ブランチには自動的に統合されない[5],[6]。

#### pull
`git fetch`と`git merge`を同時に行うもの。そのため、リモートの「master」ブランチから、ローカルの「origin/master」ブランチを介して、ローカルの「master」まで一気に最新情報を持ってくることができる[5]。

#### merge
特定のブランチ（例えばmainブランチなど）からの変更を現在作業しているブランチに統合して、新しいコミットを作成するもの[7]。

#### rebase
自分のローカルブランチのコミットをリモートの最新変更の上に再適用するもの。rebaseは履歴を再構成し、変更を一直線に整理するが、他の開発者の履歴が上書きされて消えるため、扱いには注意が必要[6]。

### 4-3. プロジェクト進行に、Pull Requestを用いたGitHub Flowを採用することで、解決できるチーム開発での問題はなにか
Pull Requestを用いたGitHub Flowを採用することで、masterブランチのデプロイ（本番環境で使える状態にする）が出来ない状態を回避することができる[1]。

masterブランチのデプロイを実現にするために、6つのルールが定められている[2]。
#### GitHub Flowの6つのルール
1. masterブランチのものは何であれデプロイ可能である。
2. masterから説明的なブランチを作成する。
3. 名前をつけたブランチに定期的にpushする。
4. いつでもプルリクエストを作る。
5. マージはプルリクエストがレビューされた後だけ。
6. レビューのあとは直ちにデプロイする。

GitHub Flowを用いた作業手順は以下の通りである[3]。
#### GitHub Flowを用いた作業手順
1. **ブランチの作成**: mainブランチから新たなブランチを作成する。  
2. **変更のコミット**: 作業ブランチ上で変更を行い、適切なコミットメッセージを付けてコミットする。  
3. **プルリクエストの作成**: 作業が完了したら、mainブランチに対してプルリクエストを作成する。  
4. **コードレビュー**: チームメンバーによるコードレビューを受ける。必要に応じてフィードバックを反映し、修正を行う。  
5. **マージとデプロイ**: レビューが承認されたら、プルリクエストをmainブランチにマージする。他の開発者のマージと衝突するなどのトラブルを避けるためにも、masterブランチへマージを実行後、すぐにデプロイを行う[3]。
6. **ブランチの削除**: 不要になったブランチは削除し、リポジトリを整理する。
